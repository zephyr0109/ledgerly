# Ledgerly 프로젝트 가이드 (GEMINI.md)

본 문서는 `ledgerly` 프로젝트의 핵심 원칙, 개발 표준 및 가이드라인을 정의합니다. 본 프로젝트의 모든 개발 활동은 이 문서에 명시된 지침을 최우선으로 준수해야 합니다.

## 1. 핵심 원칙

### 1.1 클린 아키텍처 (Clean Architecture)

이 프로젝트는 **클린 아키텍처**를 지향합니다. 코드의 각 레이어는 역할에 맞게 엄격히 분리되어야 하며, 의존성은 항상 안쪽(도메인 로직)으로 향해야 합니다.

- **Domain (Entity):** 비즈니스 규칙과 핵심 데이터 모델이 위치합니다. 프레임워크나 외부 라이브러리에 의존하지 않습니다.
- **Use Cases (Service):** 애플리케이션 특정 비즈니스 로직을 구현합니다. 도메인 객체를 조작하며, 인터페이스를 통해 외부(DB, UI)와 소통합니다.
- **Adapters (Interface Adapters):** 도메인 및 유스케이스와 외부 데이터 형식 간의 변환을 담당합니다. (예: DB 저장소 구현체, API 컨트롤러 등)
- **Infrastructure:** 데이터베이스, 파일 시스템, 외부 API 등 구체적인 구현 기술이 위치합니다.

### 1.2 기능의 멱등성 (Idempotency) 보장

모든 기능은 **멱등성**을 유지해야 합니다.

- 동일한 작업을 여러 번 수행하더라도 결과는 동일해야 합니다. (예: 동일한 엑셀 파일 중복 로드 시 데이터 중복 방지)
- 새로운 기능을 추가할 때 기존의 기능이 정상적으로 작동함을 보장해야 합니다. (회귀 테스트 필수)

### 1.3 테스트 중심 개발 (Testing)

테스트는 프로젝트의 안정성을 보장하는 가장 중요한 도구입니다.

- **단위 테스트 (Unit Test):** 각 함수와 클래스의 동작을 검증합니다.
- **통합 테스트 (Integration Test):** 여러 모듈이 함께 작동하는 과정을 검증합니다.
- **테스트 자동화:** 코드 변경 시 항상 테스트를 실행하여 기능의 정상 작동 여부를 확인합니다.
- `tests/` 디렉토리에 체계적으로 테스트 코드를 작성합니다.

## 2. 개발 및 문서화 표준

### 2.1 언어 및 주석

- **기본 언어:** 대부분의 문서(`GEMINI.md`, `README.md` 등)와 코드 내 주석은 **한국어**로 작성합니다. 또한 cli의 응답 또한 한국어로 출력합니다.
- **코드 명명:** 변수명, 함수명, 클래스명 등은 업계 표준에 따라 영어로 작성하되, 해당 코드의 의도를 설명하는 주석은 한국어로 상세히 작성합니다.

### 2.2 디렉토리 구조

- `src/ledgerly/domain`: 핵심 도메인 모델
- `src/ledgerly/usecases`: 비즈니스 로직
- `src/ledgerly/infrastructure`: DB, 파일 처리 등 외부 기술 구현
- `src/ledgerly/interfaces`: 외부 노출 인터페이스 (CLI, API 등)

## 3. 에이전트 운용 전략

- **복잡한 작업의 분리:** 대규모 리팩토링이나 복잡한 기능 구현 시 `generalist` 또는 `codebase_investigator`를 적극 활용하여 작업을 세분화합니다.
- **테스트 전담 가이드:** 테스트 코드 작성 및 검증을 위해 별도의 `SKILL.md`를 생성하여 운영할 수 있습니다.
