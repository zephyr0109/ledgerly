# Ledgerly 테스트 가이드 (SKILL.md)

이 문서는 Ledgerly 프로젝트에서 테스트를 수행하고 검증하는 방법에 대한 전문적인 지침을 제공합니다.

## 1. 테스트 목적
- 모든 기능의 **멱등성(Idempotency)** 보장.
- 새로운 기능 추가 시 기존 기능의 **회귀 방지(Regression Prevention)**.
- **클린 아키텍처** 원칙에 따른 각 레이어의 독립적인 검증.

## 2. 테스트 환경 및 도구
- **프레임워크:** `pytest`를 기본 테스트 프레임워크로 사용합니다.
- **도구:**
  - `pytest-mock`: 외부 의존성(DB, 파일 시스템)을 모킹(mocking)하여 단위 테스트 수행.
  - `pytest-cov`: 테스트 커버리지 측정.

## 3. 테스트 계층 구조

### 3.1 단위 테스트 (Unit Tests)
- **대상:** `domain/`, `usecases/` 내의 개별 함수 및 클래스.
- **위치:** `tests/unit/`
- **원칙:**
  - 하나의 테스트는 하나의 기능만 검증합니다.
  - 외부 의존성은 모두 모킹 처리합니다.
  - 도메인 로직에 대한 테스트는 100% 커버리지를 목표로 합니다.

### 3.2 통합 테스트 (Integration Tests)
- **대상:** `infrastructure/` (DB, 파일 로드) 및 여러 컴포넌트 간의 상호작용.
- **위치:** `tests/integration/`
- **원칙:**
  - 실제 DB(또는 테스트용 SQLite 파일)와 파일 시스템을 사용하여 데이터 흐름을 검증합니다.
  - 멱등성을 검증하기 위해 동일한 데이터를 두 번 처리하는 테스트 케이스를 포함합니다.

## 4. 테스트 작성 절차 (Workflow)
1. **사전 조건(Given):** 테스트에 필요한 초기 데이터나 환경을 설정합니다.
2. **동작(When):** 검증하고자 하는 기능을 실행합니다.
3. **검증(Then):** 예상된 결과가 실제 결과와 일치하는지 확인합니다. (Assertion)

## 5. 실행 및 리포팅
- 전체 테스트 실행: `pytest tests/`
- 커버리지 리포트: `pytest --cov=src/ledgerly tests/`

## 6. 에이전트 전용 가이드
- 기능을 구현할 때는 항상 대응되는 테스트 코드를 함께 제출해야 합니다.
- 테스트가 실패하는 경우, 구현 코드의 결함인지 테스트 코드의 오류인지 먼저 분석한 후 수정합니다.
- 멱등성 보장을 위해 `if_exists="append"` 등 데이터 중복 삽입 방지 로직을 철저히 테스트합니다.
